- **2026-06-24 External-app hosts stopped hand-maintaining the extapps registration table.** `tcl_maya.py` and `tcl_blender.py` each carried a per-app `(name, module, entry)` loop that re-registered extapps panels — and the two had drifted from each other and from extapps' entry points (Maya listed 7, Blender 9, extapps declares 10; `unity_workflow` was wired in neither). Both loops are deleted: uitk's `ExternalAppHandler.discover()` auto-registers every panel from extapps' entry points with `install_spec` derived from the distribution, so the hosts hold no app list. `extapps` is **kept optional/discovered, not a hard dependency** — it sits outside the `push.ps1` ecosystem release chain (`pythontk → uitk → mayatk → tentacle`), so a hard pin would escape dependency sync/cascade and transitively drag in off-chain `unitytk`; its panels appear whenever it's present in the host env. Per-host curation rides the entry-point gates + each host's existing `context_tags`: Maya (`{"maya"}`) hides the `hide_maya`-gated Substance/Marmoset panels (native bridges supersede them) and gains `unity_workflow`; Blender (`{"blender"}`) lists all 10. tentacle declares `extapps` only as a `uitk.external_app_providers` entry point — *runtime metadata, not a pip pin* (invisible to `push.ps1`) — so the handler can install-on-demand on a standalone host or self-heal if extapps is missing (suppressed inside a DCC binary, so it never wedges). The obsolete `TestExtappsRegistration` (which pinned the now-deleted table to entry points) is rewritten to guard the new contract: no host may reintroduce a manual table, and the entry points must keep the `hide_maya` gates. `test_package.py` 8/8; structural suites green.

- **2026-06-17 Fix — cameras/editors chord-menu upper+lower regions now reveal + launch their submenus over Blender (`tcl_blender` MouseTracking pump + physical button source).** A chord menu (`F12+LMB` → cameras, `F12+MMB` → editors) opens with the mouse grabbed, and the nav buttons in those menus live inside `visible_on_mouse_over` `Region`s that only reveal on hover. Two GHOST-specific facts broke that reveal: (1) the shared `MouseTracking.track()` — which synthesizes the enter events a grabbed Region needs — is gated on "a button is held", read via `QApplication.mouseButtons()`, which is **blind to GHOST's mouse**; and (2) GHOST never delivers `MouseMove` to the grabbed Qt overlay, so the menu's own event-driven tracking can't fire at all. So the regions stayed hidden and the submenus "didn't launch" (and the stale hover/grab state read as "click buttons more than once"). Fixes: `TclBlender._host_mouse_buttons()` overrides the new uitk `MouseTracking` hook to read the **physical** button state (`_KeymapBridge.physical_mouse_buttons()` via `GetAsyncKeyState`, shared with the held-button poller — DRY); and the activation poller now **drives `track()` on its timer while a chord is active** (`_KeymapBridge.chord_active`, set in `drive_press`/cleared in `drive_release`), since the timer fires independently of GHOST's mouse delivery. Live-verified on a fresh Blender via `test/blender/cameras_track_check.py` (region reveals, hover opens `cameras#lower#submenu`). Pairs with the uitk `MouseTracking(buttons_provider=…)` / `MarkingMenu._host_mouse_buttons` hook (default = Qt, so Maya is unaffected).
