- **2026-06-09 Feature** — Materials module now exposes the mayatk **Arnold Bridge** panel. Added an `Arnold Bridge` leaf to the **Tools ▸ Conversion** list (`MaterialsSlots._TOOLS_ITEMS`, next to Game Shader) and a `MaterialsSlots.b026` slot that does `marking_menu.show("arnold_bridge")` — same launcher pattern as Game Shader (`b009`) etc. The panel ships in mayatk (`mat_utils/arnold_bridge.ui` + `ArnoldBridgeSlots`); tentacle only launches it. - **2026-06-07 Bugfix** — `uv.py b011` (Sew UVs) crashed with `# RuntimeError: # No object matches name: <shape>` when the selection held two transforms sharing a short leaf name (their shapes share a short name). It resolved each shape with `cmds.listRelatives(shapes=True, noIntermediate=True)` (short names) then gated on `cmds.objectType(shape)` — and an ambiguous short shape name resolves to *nothing*, so objectType raised. Now `listRelatives(..., type="mesh", fullPath=True)` returns unambiguous full-path mesh shapes directly (the fragile per-shape `objectType` call is gone). Covered by `test/test_uv.py::TestB011SewUVsDuplicateNames` (two `dupCyl` cylinders under different groups: old code raised, fix sews both by full path).
- **2026-06-07 Feature** — `uv.py tb009` button renamed **Unwrap Cylinder → Cut Cylinder** (`uv.ui` + `uv#submenu.ui` text/tooltip, `tb009_init` title/docstrings) to reflect that it cuts the seams and the **Unfold** step (`chk041`, default on) is optional — unchecking it cuts the cap/step/lengthwise seams only. No behavior change to the seam cut or `UvUtils.unwrap_cylinder(unfold=…)` call; the option already existed.
- **2026-06-07 Quality (upstream)** — **Cut Cylinder** (`tb009`) and the **Unfold** auto-cylinder branch (`tb004`) now unwrap **low-poly** tubes cleanly: an 8-sided cylinder no longer shatters on its 45° faceting, single-row bands no longer collapse, and shells are packed in 0-1 without mirroring (all fixed in `mtk.UvUtils.unwrap_cylinder` — see mayatk CHANGELOG). No tentacle code change; verified against the A/B reference scene (case A reproduces the hand-authored 9-shell target exactly).
- **2026-06-07 Bugfix/Feature** — `uv.py tb004` (Unfold) did nothing on a seamless mesh and, once seamed, the unfold collapsed the UVs. Two parts: (1) **Auto-seam** — a closed mesh in a single UV shell has no borders for `u3dUnfold` to open along, so Unfold now detects that (`_mesh_has_unfold_seams`: >1 UV shell, or any 3D-boundary edge ⇒ unfoldable as-is; else route to auto-unwrap) and routes such meshes through `mtk.UvUtils.unwrap_cylinder` (hard-crease + lengthwise-column seams, then unfold). Meshes that already have seams (or open tubes) take the normal relax path untouched, so existing UV work isn't destroyed. (2) **Collapse fix (upstream)** — `unwrap_cylinder`'s unfold used to collapse every shell to a point when the source carried a degenerate axis-aligned projection; mayatk now seeds each shell with a thinnest-axis planar projection before `u3dUnfold` and packs into 0-1 (see mayatk CHANGELOG). Verified against the reference scene: a seamless turned mesh now unfolds to 9 clean, non-overlapping shells. Covered upstream by `test_uv_utils.py::test_unwrap_unfold_does_not_collapse`.
- **2026-06-07 Feature** — `uv.py tb009` (Unwrap Cylinder) option box gained a **Crease Angle** spinbox (`s016`, 1–179°, default 45) wired to `UvUtils.unwrap_cylinder(angle=…)` — the threshold for how sharp a bend must be to start a new shell, so a smaller value splits at gentler bevels and a larger value keeps only the hardest (~90°) steps. The angle already existed on the mtk API; the tool no longer hardcodes the default.
- **2026-06-06 Feature** — `uv.py tb009` (Unwrap Cylinder) is now a single clean auto-unwrap (the short-lived **Seam Mode** combobox was removed). It drives mayatk's rewritten `UvUtils.unwrap_cylinder`, which cuts the hard creases (cap rims + ~90° step rings) plus one lengthwise seam and unfolds — so a turned / stepped profile peels into one UV shell per smooth section, each flat step, and each cap, while shallow chamfers stay merged. Validated against a real reference scene (a turned baluster reproduces the hand-authored 9-shell target exactly). The option box keeps just **Invert Seam** / **Unfold** / **Orient**. Covered upstream by `mayatk/test/test_uv_utils.py::TestUvCylinderUnwrap`.
- **2026-06-06 Bugfix** — `uv.py tb004` (Unfold) **`selectMode` guard was inverted** (`if cmds.selectMode(query=True, object=True)` → `if not …`): contrary to its own comment it never switched out of component mode, so a leftover component selection (normal mid-UV-edit) could scope `u3dUnfold` to a partial sub-shell or change its behavior between clicks — the likely cause of *Repair + Retry* appearing to need two clicks (forcing object mode up front lets the single repair retry land in one click). The reactive try → fail → repair → retry flow is **retained**: an earlier attempt to make it proactive (pre-scan with `polyInfo` and abort/repair *before* unfolding) **regressed unfold to fail on every mesh** — `polyInfo`'s non-manifold-vertex flag is broader than what `u3dUnfold` actually rejects, so meshes Unfold3D handles fine were wrongly aborted in the default Warn + Select mode. So u3dUnfold stays the sole authority on unfoldability; we only warn/repair on its real `RuntimeError`. `test/test_uv.py`: `TestTb004NonManifoldStrategy.test_does_not_preempt_unfold_on_polyinfo_flag` (regression — polyInfo-flagged mesh that u3dUnfold accepts must still unfold, no warn/abort), `test_repair_mode_repairs_and_retries` (geometry-driven stub: raises only while still non-manifold), and `TestTb004ObjectModeGuard` (spies `selectMode`; fails on the old inverted guard). Verified under mayapy (Unfold3D loaded): one `clean_geometry` pass clears the non-manifold case and the retry succeeds.
- **2026-06-06 Bugfix** — `transform.py tb005` (Move To) **Match Scale** option scaled each axis independently — `mtk.match_scale(source, target)` defaults to per-axis bounding-box ratios (`bx/ax, by/ay, bz/az`), distorting the source to fit the target's box exactly. Now passes `average=True` so the moved object(s) are scaled by a single uniform factor (the averaged ratio), preserving aspect. Tooltip updated to say "uniformly … (no per-axis distortion)". The standalone **Match Scale** button (`b001`) is unchanged (still per-axis). `test/test_transform.py::TestTb005MoveTo.test_match_scale_enabled` now asserts `average=True` is forwarded; verified under mayapy standalone (uniform 3.8333 across all axes on mismatched cubes).

Its option box (`tb009_init`) exposes **Invert Seam** (`chk040`, land the lengthwise seam on the opposite/hidden side), **Unfold** (`chk041`), and **Orient** (`chk042`); map size comes from the panel's resolution combo. - **2026-06-04 Feature** — `uv.py tb004` (Unfold) gained a **non-manifold strategy** combobox (`cmb013`, option box) for what to do when non-manifold geometry blocks Unfold: **Warn + Select** (default) or **Repair + Retry**. *Warn + Select* selects the offending vertices (switching to vertex component mode via `selectMode`/`selectType`) and shows a formatted message explaining the cause and fixes. *Repair + Retry* auto-cleans the geometry via `mtk.Diagnostics.clean_geometry(repair=True, nonmanifold=True)` (Mesh Cleanup) and runs Unfold again; on success it proceeds through optimize/orient/stack and shows a brief result message noting the repair (vertices fixed), with a per-mesh breakdown printed to the Script Editor; if the mesh is still unfoldable-blocking it falls back to Warn + Select. Detection uses native `cmds.polyInfo(shape, nonManifoldVertices=True)` (instant) rather than the per-vertex Python scan in `EditUtils.find_non_manifold_vertex` (too slow on heavy meshes). Helpers `UvSlots._non_manifold_vertices` (guards empty input — `cmds.ls([])` matches all scene meshes), `_warn_and_select_non_manifold`, `_repair_non_manifold`. Tests in `test/test_uv.py` (`TestTb004NonManifoldStrategy`) drive the real polyInfo/clean_geometry/selection paths against a bowtie mesh (select, repair+retry, and unrepairable-fallback); verified under mayapy standalone.
- **2026-06-04 Bugfix** — `uv.py tb004` (Unfold) crashed with an unhandled `RuntimeError: Mesh has non-manifold vertices` straight out of `cmds.u3dUnfold` when the selected mesh was non-manifold. - **2026-06-02 Bugfix** — `tcl_maya.py` registered every extapps app with `module=f"extapps.{_name}"`, but `metashape_workflow` lives in a subpackage (`extapps.photogrammetry.metashape_workflow`). The derived path was wrong and **overwrote** the correct one auto-discovery had registered (the manual call wins, carrying `install_spec`), so launching the Metashape panel hit `_is_importable() == False` → surprise `pip install` → *"Install of 'extapps' completed but 'extapps.metashape_workflow' is still not importable"*. Fixed by spelling out each app's `module` explicitly in the registration table instead of deriving it from the name (the name→module mapping isn't 1:1 once an app moves into a subpackage). uitk's `ExternalAppHandler.launch()` self-heal masked this whenever the entry-point metadata was present and correct, but not on a machine where `extapps` wasn't yet installed — this fixes the root cause. Verified in a fresh Maya (`mayatk.MayaConnection`): the corrected registration is importable on the first check (no self-heal needed) and the panel constructs.
