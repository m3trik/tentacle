- **2026-06-12 Fix (diagnostics)** — Made the Blender activation **self-report through user-visible channels**, after a live session repeatedly showed "F12 still opens render, nothing prints." Root cause of the *invisible* part: a keymap-invoked operator's `print()` goes to the hidden **system** console (Window ▸ Toggle System Console), never the **Python Console panel** — so the `_DEBUG` "key fired" log the user was told to watch for could never appear there even if the operator fired. Fixes: (1) the bridge operator now reports via `self.report({'INFO'/'WARNING'}, ...)` (status bar / Info editor, actually visible) instead of `print()` under `_DEBUG`, distinguishing "fired → show()" from "fired but no live menu → render"; (2) `diagnose()` **returns** its report string (shown directly in the Python Console panel) with a plain-language **VERDICT** (`LIKELY WORKING` / `PROBLEM: …` / `CONFLICT: …`) and surfaces the live `module file` (stale/duplicate `tentacle` on `sys.path` is the #1 silent cause); (3) `register()` returns that report and raises a Blender **popup** on any PROBLEM/CONFLICT, so a console-blind user still gets feedback; (4) rival-detection now scans the **addon** keyconfig too (another add-on binding bare F12 in the 3D View region is a real conflict the clean factory session can't show). The unverifiable link remains a *physical* keystroke: it can't be auto-injected here (Blender launched from a background shell is denied OS foreground for `keybd_event`, and GHOST takes keyboard via Raw Input/`WM_INPUT`, which `PostMessage` can't forge — `gui_keypress_check.py` rewritten to `PostMessage` confirms the wall). GUI dump still verifies install is correct (`3D View`/F12 active, no rivals, `module file` right) and both `show()` + operator-invoke surface the menu; dead `_dispatch_keyconfigs` removed. Headless 17/17, structural 9/9, GUI verdict `LIKELY WORKING`.
- **2026-06-12 Fix/Simplify** — Blender activation reverted to a **viewport-scoped** keymap; the global-keymap + global-mute approach was over-engineered. GUI dump confirms our item sits in `3D View` (active) with render left untouched, and both `show()` and operator-invoke surface the menu. The synthetic-conflict mute checks were dropped from the harnesses; `keymap_bridge_check.py` instead asserts the keymap is `3D View`-scoped. (The prior global-`Window`+mute entry below is superseded.)
